# Referência de Frameworks de Ideação

Use estes frameworks com seletividade. Escolha a lente que se encaixa na ideia — não rode mecanicamente todo framework. O objetivo é desbloquear o pensamento, não seguir um checklist.

## SCAMPER

Uma forma estruturada de transformar uma ideia existente aplicando sete operações diferentes:

- **Substituir:** Qual componente, material ou processo você poderia trocar? E se substituísse a tecnologia central? O público-alvo? O modelo de negócio?
- **Combinar:** E se fundisse isso com outro produto, serviço ou ideia? Quais duas coisas que normalmente não combinam criariam algo novo?
- **Adaptar:** O que mais é parecido com isso? Quais ideias de outras indústrias, domínios ou períodos você poderia emprestar? Existe um paralelo na natureza?
- **Modificar (Ampliar/Minimizar):** E se tornasse 10x maior? 10x menor? E se exagerasse uma funcionalidade? E se reduzisse ao mínimo absoluto?
- **Propor outros usos:** Quem mais poderia usar isso? Quais outros problemas poderia resolver? O que acontece se usar em um contexto completamente diferente?
- **Eliminar:** O que acontece se remover uma funcionalidade inteiramente? Qual é a versão com zero configuração? Como seria com metade dos passos?
- **Reverter/Reorganizar:** E se fizesse os passos na ordem oposta? E se o usuário fizesse o trabalho em vez do sistema (ou vice-versa)? E se revertesse a cadeia de valor?

**Melhor para:** Melhorar ou reimaginationar produtos/funcionalidades existentes. Menos útil para ideias do zero.

## Como Poderíamos (How Might We - HMW)

Reenquadre problemas como oportunidades usando o formato "Como poderíamos...":

- Comece com uma observação ou ponto de dor
- Reenquadre como "Como poderíamos [resultado desejado] para [usuário específico] sem [restrição chave]?"
- Gere múltiplos reenquadramentos HMW do mesmo problema — reenquadramentos diferentes desbloqueiam soluções diferentes

**Qualidades de um bom HMW:**
- Estreito o suficiente para ser acionável ("...ajudar novos usuários a encontrar conteúdo relevante nos primeiros 5 minutos")
- Amplo o suficiente para permitir soluções criativas (não "...adicionar uma barra lateral de recomendações")
- Contém uma tensão ou restrição que força criatividade

**Qualidades de um HMW ruim:**
- Muito amplo: "Como poderíamos deixar os usuários felizes?"
- Muito estreito: "Como poderíamos adicionar um botão na página de configurações?"
- Solução embutida: "Como poderíamos construir um chatbot para suporte?"

**Melhor para:** Reenquadrar pensamento travado. Quando alguém está ancorado em uma solução, puxe de volta para o problema.

## Pensamento por Primeiros Princípios

Desmonte a ideia até suas verdades fundamentais, depois reconstrua a partir daí:

1. **O que sabemos que é verdade?** (não assumido, não convencional — realmente verdade)
2. **O que estamos assumindo?** Liste toda suposição, até as que parecem óbvias
3. **Quais suposições podemos desafiar?** Para cada uma, pergunte: "Isso é realmente uma lei da física ou apenas como sempre foi feito?"
4. **Reconstrua a partir das verdades.** Se tivesse apenas as verdades fundamentais, o que construiria?

**Melhor para:** Sair do pensamento incremental. Quando toda ideia parece uma pequena melhoria do status quo.

## Jobs to Be Done (JTBD)

Foque no que o usuário está tentando realizar, não no que ele diz que quer:

- **Trabalho funcional:** Qual tarefa estão tentando completar?
- **Trabalho emocional:** Como querem se sentir?
- **Trabalho social:** Como querem ser percebidos?

Formato: "Quando eu [situação], quero [motivação], para poder [resultado esperado]."

**Insight chave:** Pessoas não compram produtos — elas os contratam para fazer um trabalho. O produto concorrente nem sempre está na mesma categoria. (Netflix compete com o sono, não apenas com outros serviços de streaming.)

**Melhor para:** Entender o problema real. Quando não tem certeza se está resolvendo a coisa certa.

## Ideação Baseada em Restrições

Impor restrições deliberadamente para forçar soluções criativas:

- **Restrição de tempo:** "E se tivesse apenas 1 dia para construir isso?"
- **Restrição de funcionalidade:** "E se pudesse ter apenas uma funcionalidade?"
- **Restrição de tecnologia:** "E se não pudesse usar [a tecnologia óbvia]?"
- **Restrição de custo:** "E se tivesse que ser grátis para sempre?"
- **Restrição de público:** "E se seu usuário nunca tivesse usado um computador?"
- **Restrição de escala:** "E se precisasse funcionar para 1 bilhão de usuários? E para apenas 10?"

**Melhor para:** Cortar complexidade. Quando a ideia está crescendo demais ou ficando vaga demais.

## Pré-mortem

Imagine que a ideia já falhou. Trabalhe de trás para frente:

1. Daqui a 12 meses. O projeto foi lançado e fracassou. O que deu errado?
2. Liste toda razão plausível de falha — técnica, de mercado, de equipe, de timing
3. Para cada modo de falha: Isso é prevenível? Isso é um sinal de que a ideia precisa mudar?
4. Quais modos de falha você está disposto a aceitar? Quais matariam o projeto?

**Melhor para:** Avaliação na Fase 2. Testar sob pressão ideias que parecem boas mas não foram pressionadas.

## Inspiração Análoga

Olhe como outros domínios resolveram problemas similares:

- Qual indústria já resolveu uma versão desse problema?
- Como seria isso se [empresa/produto específico] tivesse construído?
- Qual sistema natural funciona dessa forma?
- Qual precedente histórico existe?

A chave é encontrar similaridades *estruturais*, não superficiais. "Uber para X" é superficial. "Um marketplace de dois lados que resolve um problema de confiança entre estranhos" é estrutural.

**Melhor para:** Expansão na Fase 1. Gerar variações que parecem genuinamente diferentes da abordagem óbvia.
